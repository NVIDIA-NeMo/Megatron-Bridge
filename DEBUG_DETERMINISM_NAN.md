# Root Cause Analysis: NaN in LLaMA 3.1 405B Deterministic Mode on GB200

## Summary

When training LLaMA 3.1 405B with `model.deterministic_mode=true` on GB200, gradient norms are NaN from the very first iteration. This is **not** a flash attention bug, not an arithmetic precision issue, and not a backward-pass problem. It is a **GPU memory race condition** caused by the interaction of three features:

1. **`overlap_p2p_comm=True`** — async (non-blocking) NCCL P2P sends between PP stages
2. **`deallocate_pipeline_outputs=True`** — immediately free the output tensor buffer after initiating the send
3. **`TORCH_NCCL_AVOID_RECORD_STREAMS=1`** (now default in PyTorch NCCL) — tells NCCL not to pin the tensor's CUDA stream during an async send, requiring the caller to manually call `.wait()` before the buffer is freed

NCCL itself explicitly warns about this in the log. The NaN enters PP stage 8's input from PP stage 7, not because PP stage 7 computed a wrong value, but because the GPU buffer holding the correct value was freed and overwritten **while NCCL was still in the middle of transmitting it**.

## What the Logs Show

### Step 1 — NCCL Issues the Exact Warning

At log line 7284 (timestamp 02:27:54), right as PP stage 7 (ranks 60–63) enters the 1F1B overlap phase and initiates its first async inter-stage send, NCCL immediately prints:

```
[rank61]:[W320 02:27:54.752717857 ProcessGroupNCCL.cpp:580] Warning:
    TORCH_NCCL_AVOID_RECORD_STREAMS=1 is experimental for point-to-point
    collectives. To ensure safety, .wait() must be called on all returned
    handles before they fall out of scope, including for isend() calls.
[rank62]:[W320 02:27:54.765418329 ...] (same warning)
[rank63]:[W320 02:27:54.771285781 ...] (same warning)
[rank60]:[W320 02:27:54.793228678 ...] (same warning)
```

This warning is NCCL's own documentation of the unsafe condition. It fires once per `isend()` that is issued while `TORCH_NCCL_AVOID_RECORD_STREAMS=1`. Every PP stage pair logs this warning as it enters 1F1B mode.

### Step 2 — The NaN Appears at PP=8 Input (Log Line 8280)

About 28 seconds later, after processing microbatches 0–30 cleanly, microbatch 31 (the last microbatch in the 2nd group) is NaN when it arrives at PP stage 8:

```
[SCHED_NAN] rank=68 pp=8 virtual_mb=143 model_chunk=0 microbatch=31
    input_tensor dtype=torch.bfloat16 shape=(2048, 1, 16384)
```

This fires from the `[SCHED_NAN]` probe placed at the top of `forward_step_helper()` in `schedules.py`. It means: before PP stage 8's first transformer layer has even started processing this microbatch, the input activation is already fully NaN.

### Step 3 — NaN Floods All Sublayers at PP=8 (Log Lines 8281–8287)

```
[NAN_TRACK] rank=68 pp=8 layer=8 input_hidden:      nan=True ... grad_enabled=False
[NAN_TRACK] rank=68 pp=8 layer=8 after_norm1:       nan=True ... grad_enabled=False
[NAN_TRACK] rank=68 pp=8 layer=8 after_attn:        nan=True ... grad_enabled=False
[NAN_TRACK] rank=68 pp=8 layer=8 after_attn_residual: nan=True ...
[NAN_TRACK] rank=68 pp=8 layer=8 after_norm2:       nan=True ...
[NAN_TRACK] rank=68 pp=8 layer=8 after_mlp:         nan=True ...
[NAN_TRACK] rank=68 pp=8 layer=8 after_mlp_residual: nan=True ...
```

`grad_enabled=False` indicates this is inside `CheckpointFunction.forward()` running under `torch.no_grad()` — the normal forward pass, not a backward recompute. Every single sublayer output is NaN because the **input** was already NaN. None of the sublayers introduced the NaN.

### Step 4 — PP=7 Layer 7 Is Completely Clean

The `[NAN_TRACK]` probe prints **only when NaN is detected**. There are zero NAN_TRACK messages for `pp=7 layer=7` in the entire log. The first NAN_TRACK for pp=7 appears at:

```
[NAN_TRACK] rank=60 pp=7 layer=23 input_hidden: nan=True ... (log line 8401)
```

Layer=23 is a different VPP chunk processed later — it received NaN **from PP=8** coming back via the pipeline schedule, not something generated by PP=7 itself. PP=7 layer=7 computed a valid output for microbatch=31 and sent it cleanly.

**Conclusion: the NaN was not in the data that PP=7 sent. It was introduced in transit.**

### Step 5 — NaN Cascades Through All Downstream PP Stages

Immediately after PP=8 (log line 8288):

```
[SCHED_NAN] rank=76 pp=9  virtual_mb=143 model_chunk=0 microbatch=31  (line 8288)
[SCHED_NAN] rank=84 pp=10 virtual_mb=143 model_chunk=0 microbatch=31  (line 8296)
[SCHED_NAN] rank=92 pp=11 virtual_mb=143 model_chunk=0 microbatch=31  (line 8304)
...
```

Every subsequent PP stage receives NaN for this microbatch. The corruption propagates all the way to the last PP stage and into the loss.

### Step 6 — Checkpoint Recompute Inherits NaN, Does Not Create It (Log Lines 9461+)

During the backward pass, `CheckpointFunction` recomputes the forward to recover activations. Our probe in `tensor_parallel/random.py` checks the saved tensor before recompute:

```
[CKPT_RECOMPUTE] rank=116 SAVED TENSOR NaN BEFORE RECOMPUTE:
    nan=True inf=False dtype=torch.bfloat16 shape=(2048, 1, 16384)  (line 9461)
[CKPT_RECOMPUTE] rank=116 OUTPUT NaN AFTER RECOMPUTE (INHERITED FROM INPUT):
    dtype=torch.bfloat16 shape=(2048, 1, 16384)  (line 9481)
```

The saved tensor was already NaN before recompute began. Recompute ran fine on NaN input and produced NaN output — as expected (NaN propagates through valid arithmetic). The recompute did not create or amplify the NaN; it inherited it.

### Step 7 — SCHED_NAN_RECV Appears Only in Iteration 2+ Warmup (Log Line 19220+)

```
[SCHED_NAN_RECV] rank=12 pp=1 warmup_k=0 routed_to_chunk=0
    NaN in tensor just received from prev PP stage  (line 19220)
[SCHED_NAN_RECV] rank=12 pp=1 warmup_k=1 routed_to_chunk=0  (line 19242)
```

The `[SCHED_NAN_RECV]` probe was placed only in the **blocking warmup recv path**. The 1F1B overlap path (`pp_post_forward`) uses async recv and was not instrumented with this probe. That is why there is no `SCHED_NAN_RECV` at iteration 1 microbatch=31 — the NaN crossed via the async 1F1B path, not the warmup blocking path.

## Root Cause: Async Send + Immediate Dealloc Race Under AVOID_RECORD_STREAMS

### The three pieces that interact

**`overlap_p2p_comm=True`** (schedules.py `pp_post_forward`):
After each forward step in the 1F1B steady state, the schedule calls:
```python
send_handle = send_forward_recv_backward(output_tensor, ...)  # async NCCL isend
deallocate_output_tensor(output_tensor, config)               # frees GPU buffer IMMEDIATELY
```

**`deallocate_output_tensor`** (schedules.py):
```python
def deallocate_output_tensor(out, deallocate_pipeline_outputs):
    if deallocate_pipeline_outputs:
        out.untyped_storage().resize_(0)  # frees the GPU buffer
```

**`TORCH_NCCL_AVOID_RECORD_STREAMS=1`** (now the PyTorch default):
Normally, when you issue an async NCCL operation on a CUDA tensor, NCCL records the CUDA stream used by the operation into the tensor's allocator metadata. This prevents the allocator from reusing that memory until the stream completes. With `AVOID_RECORD_STREAMS=1`, NCCL skips this recording — it assumes the caller will call `.wait()` on the returned handle before freeing the buffer.

### The race condition

```
Timeline on PP=7 (sender):

T1: Forward pass for microbatch=31 completes.
    output_tensor = valid_activations (shape [2048, 1, 16384], all finite BF16)

T2: pp_post_forward() called:
    send_handle = NCCL.isend(output_tensor → PP=8)
    # NCCL starts DMA copy of output_tensor to NIC/interconnect buffer
    # Because AVOID_RECORD_STREAMS=1, NCCL does NOT pin this memory
    # NCCL warns: ".wait() must be called before they fall out of scope"

T3: deallocate_output_tensor(output_tensor) called IMMEDIATELY:
    output_tensor.untyped_storage().resize_(0)
    # GPU allocator marks this 2048*16384*2 bytes as FREE

T4: GPU allocator reuses that freed memory for other computation
    # e.g., another microbatch's QKV projection overwrites it with new values

T5: NCCL DMA is still in progress — reads the now-overwritten memory
    # Reads garbage / partial overwrites → NaN in bfloat16

PP=8 receives corrupted data.
```

The critical issue: `deallocate_output_tensor` happens **synchronously on the CPU** immediately after `isend()` returns. `isend()` returns before the DMA copy is complete. Under normal NCCL, the stream recording would prevent the allocator from reusing memory until the DMA finishes. Under `AVOID_RECORD_STREAMS=1`, that protection is removed — and the code never calls `send_handle.wait()` before deallocating.

### Why only microbatch=31?

Microbatch=31 is the last microbatch in the 2nd group of 16 (groups are indexed 0–1 across `microbatch_group_size_per_vp_stage=16`). At this point the pipeline is deep into 1F1B steady state and has high memory pressure — many microbatches in flight simultaneously across 16 PP stages × 8 VPP chunks. The GPU allocator is under the most pressure to reuse freed memory quickly. Earlier microbatches (0–30) may have been "lucky" — the freed buffer was not reused within the NCCL DMA window. Microbatch=31 was unlucky.

### Why only GB200, not H100 at smaller scale?

GB200 is significantly faster than H100. The window between `isend()` and "NCCL finishes the DMA" is short in absolute time, but the window between `deallocate_output_tensor()` and "allocator reuses memory" is even shorter on GB200 because more compute happens faster. The race exists on H100 too but is less likely to be triggered within 10 steps for smaller models with fewer in-flight microbatches.

---

## Why Flash Attention Is Not the Cause

An early hypothesis was that `flash_attention deterministic=True` on SM100 (GB200) introduced numerical instability. This was ruled out:

- The NaN is in `input_hidden` before any sublayer computation begins (log line 8281)
- PP=7 layer=7 has zero NAN_TRACK entries — its output was clean
- The NaN appears at the PP boundary (between PP=7 and PP=8), not at any attention or MLP computation
- Switching to `fused` attention would not change the P2P communication path at all

---

## Why the Recompute Is Not the Cause

Another hypothesis was that `CheckpointFunction` recomputes the forward under different dtype/autocast context, introducing a discrepancy. This was ruled out:

- `[CKPT_RECOMPUTE] SAVED TENSOR NaN BEFORE RECOMPUTE` (line 9461) — the saved tensor is already NaN before recompute starts
- `OUTPUT NaN AFTER RECOMPUTE (INHERITED FROM INPUT)` (line 9481) — recompute ran on NaN input and produced NaN output, as expected
- The autocast context during recompute is irrelevant if the input itself is NaN

## Diagnostic Probes Added

### `schedules.py` — `[SCHED_NAN]`
Fires when `input_tensor` has NaN at the entrance to `forward_step_helper()`, before any computation:
```python
[SCHED_NAN] rank=68 pp=8 virtual_mb=143 model_chunk=0 microbatch=31
    input_tensor dtype=torch.bfloat16 shape=(2048, 1, 16384)
```

### `schedules.py` — `[SCHED_NAN_RECV]`
Fires in the warmup blocking-recv path when a just-received tensor has NaN.
**Note**: does NOT cover the 1F1B async-recv path — see iteration 2 warmup (line 19220).

### `transformer_layer.py` — `[NAN_TRACK]`
7 check-points per transformer layer: `input_hidden`, `after_norm1`, `after_attn`, `after_attn_residual`, `after_norm2`, `after_mlp`, `after_mlp_residual`. Only prints when NaN/Inf is present.
`grad_enabled=False` = inside `CheckpointFunction.forward()` (no_grad).
`grad_enabled=True` = inside `CheckpointFunction.backward()` recompute (enable_grad).

### `random.py` — `[CKPT_RECOMPUTE]`
Checks saved tensors before and after backward recompute. Distinguishes whether NaN was present before recompute (inherited) or introduced by it.

## The Fix

The fix must ensure the GPU buffer is not freed until the async NCCL send completes. In `schedules.py`, in the `pp_post_forward` 1F1B overlap path, the send handle must be waited on before `deallocate_output_tensor` is called:

```python
# Current (broken):
send_next_wait_handle = send_forward_recv_backward(output_tensor, ...)
deallocate_output_tensor(output_tensor, config)  # frees buffer while send is in flight

# Fixed:
send_next_wait_handle = send_forward_recv_backward(output_tensor, ...)
if send_next_wait_handle is not None:
    send_next_wait_handle.wait()  # ensure DMA completes before freeing
deallocate_output_tensor(output_tensor, config)  # now safe
```

This satisfies the requirement stated in NCCL's own warning: *".wait() must be called on all returned handles before they fall out of scope, including for isend() calls."*

An alternative (more conservative) fix is to disable `deallocate_pipeline_outputs` in deterministic mode, at the cost of higher peak memory.

## Verification Experiments

Two experiments were run to confirm the root cause:

### Test 1 — Disable `deallocate_pipeline_outputs` (job 1806932)

```bash
DETERMINISTIC=true BACKEND=flash GPU=gb200 NO_DEALLOC=true ./run_llama3_405b.sh
# adds: model.deallocate_pipeline_outputs=false
```

**Result: NaN completely eliminated.**

```
iteration 1: lm loss: 4.314195E+00 | grad norm: 1097.251 | number of nan iterations: 0
iteration 2: lm loss: 4.352941E+00 | grad norm:   46.625 | number of nan iterations: 0
iteration 3: lm loss: 3.988099E+00 | grad norm:   29.694 | number of nan iterations: 0
iteration 4: lm loss: 3.800318E+00 | grad norm:   79.467 | number of nan iterations: 0
```

Disabling deallocation removes the race — the GPU buffer stays valid until NCCL finishes. This **confirms the `isend` + `deallocate` race is the root cause**.

### Test 2 — Non-deterministic mode with `tp_comm_overlap=false` (job 1806933)

```bash
BACKEND=flash GPU=gb200 NO_TP_OVERLAP=true ./run_llama3_405b.sh
# adds: comm_overlap.tp_comm_overlap=false to non-deterministic run
```

**Result: No NaN either.**

```
iteration 1: lm loss: 4.314195E+00 | grad norm: 1095.094 | number of nan iterations: 0
iteration 2: lm loss: 4.352941E+00 | grad norm:   46.664 | number of nan iterations: 0
iteration 3: lm loss: 3.988202E+00 | grad norm:   35.173 | number of nan iterations: 0
```

`tp_comm_overlap=false` alone does not trigger the race in non-deterministic mode. Something else in the deterministic config (likely `NCCL_ALGO=Ring`, `NVTE_ALLOW_NONDETERMINISTIC_ALGO=0`, or cuBLAS workspace config) affects GPU timing enough to make the race fire. The tp_comm_overlap hypothesis is unconfirmed.
